# MPU6050 模块设计说明

## 1. 模块目标

`mpu6050` 是一个寄存器型传感器模块。它对上提供统一的初始化、状态查询、寄存器读写和原始数据读取接口；对下兼容硬件 IIC 和软件 IIC 两类 drv 实现。

这个模块文档只说明一件事：当前目录下几个文件分别负责什么，以及 `mpu6050_port.c` 里的链接函数应该怎么写，才能满足 `mpu6050.c` 的需求。

## 2. 文件分工

### 2.1 `mpu6050.h`

负责定义：

- 设备逻辑编号 `eMPU6050MapType`。
- 寄存器地址、协议常量、状态码宏。
- 配置结构 `stMpu6050Cfg`。
- 原始数据结构 `stMpu6050RawSample`。
- 运行上下文 `stMpu6050Device`。
- core 需要使用的 IIC port 接口表和绑定结构。
- 对外公共 API。

这里把 `stMpu6050PortIicInterface` 和 `stMpu6050PortIicBinding` 放在头文件中，是因为 core 和 port 都要看到这份契约。

### 2.2 `mpu6050.c`

负责实现：

- 默认配置懒加载。
- 配置合法性检查。
- `Init()` 的完整 bring-up 流程。
- `ReadReg()`、`WriteReg()`、`ReadRaw()`、`ReadTempCdC()` 等业务接口。
- `isReady` 和最近一次采样缓存维护。

它只依赖 `stMpu6050PortIicInterface` 里的三个动作：

- `init`
- `writeReg`
- `readReg`

除此之外，core 不应该直接知道底层是 `drvIic` 还是 `drvAnlogIic`。

### 2.3 `mpu6050_port.h`

负责声明：

- 默认配置和默认绑定函数。
- 绑定修改函数。
- 绑定合法性检查函数。
- 接口表完整性检查函数。
- 获取当前绑定接口表的函数。
- 毫秒延时函数。

这些函数本质上是 core 访问项目环境的唯一入口。

### 2.4 `mpu6050_port.c`

负责实现：

- 默认设备配置数组 `gMpu6050PortDefCfg`。
- 硬件 IIC / 软件 IIC 对应的 adapter 函数。
- `gMpu6050PortIicInterfaces` 接口表。
- 绑定切换函数。
- 绑定校验函数。
- FreeRTOS / 裸机共用的 `mpu6050PortDelayMs()`。

换句话说，这个文件不是在写传感器逻辑，而是在写“本工程如何把 IIC 驱动接给 MPU6050 core”。

### 2.5 `mpu6050_debug.h/.c`

负责实现：

- 可选 console 注册入口。
- `list`、`init`、`id`、`regget`、`regset`、`raw`、`temp` 等调试命令。
- 仅暴露联调和定位问题所需的最小必要能力。

## 3. core 需要的最小底层能力

从 `mpu6050.c` 的调用点反推，core 对 port 只有四类需求：

1. 能初始化指定 bus。
2. 能按寄存器地址写数据。
3. 能按寄存器地址读数据。
4. 能提供毫秒级等待。

因此接口表只保留下面三个动作：

```c
typedef struct stMpu6050PortIicInterface {
    mpu6050PortIicInitFunc init;
    mpu6050PortIicWriteRegFunc writeReg;
    mpu6050PortIicReadRegFunc readReg;
} stMpu6050PortIicInterface;
```

这是当前模块最重要的契约。只要 port 能正确实现这三个函数，core 基本就能复用到别的工程。

## 4. port 链接函数设计

### 4.1 链接函数的目标

`mpu6050_port.c` 里的 adapter 负责把底层 drv 翻译成 core 能理解的统一接口。当前有两组：

- 软件 IIC adapter。
- 硬件 IIC adapter。

它们必须做到：

- 输入参数形式和 core 的访问语义一致。
- 内部把 `uint8_t bus` 转成对应 drv 枚举。
- 在调用 drv 前完成 bus 合法性检查。
- 返回 `eDrvStatus`，让 core 直接处理失败。

### 4.2 当前模块需要的 adapter 形态

对于 MPU6050，这三类 adapter 是必须的：

```c
static eDrvStatus mpu6050PortSoftIicInitAdpt(uint8_t bus);
static eDrvStatus mpu6050PortSoftIicWriteRegAdpt(uint8_t bus, uint8_t address, const uint8_t *regBuf, uint16_t regLen, const uint8_t *buffer, uint16_t length);
static eDrvStatus mpu6050PortSoftIicReadRegAdpt(uint8_t bus, uint8_t address, const uint8_t *regBuf, uint16_t regLen, uint8_t *buffer, uint16_t length);
```

硬件 IIC 对应函数签名保持完全一致。

这样设计的原因不是形式统一，而是因为 core 内部访问寄存器时就需要这组参数：

- `bus`：选哪一路 IIC。
- `address`：器件地址。
- `regBuf` 和 `regLen`：寄存器地址段。
- `buffer` 和 `length`：读写数据段。

如果 adapter 签名比这更弱，core 就得知道更多底层细节；如果 adapter 签名比这更强，core 就会被迫依赖不必要的 drv 细节。

### 4.3 adapter 里应该做什么

adapter 内部允许做下面这些事：

- 检查 `bus` 是否越界。
- 把 `bus` 转成 `eDrvIicPortMap` 或 `eDrvAnlogIicPortMap`。
- 调用 `drvIicInit()`、`drvIicReadRegister()`、`drvIicWriteRegister()`。
- 调用 `drvAnlogIicInit()`、`drvAnlogIicReadRegister()`、`drvAnlogIicWriteRegister()`。

### 4.4 adapter 里不应该做什么

adapter 不应该做这些事：

- 不应该判断 WHO_AM_I 是否匹配。
- 不应该决定复位顺序。
- 不应该解析温度和加速度数据。
- 不应该偷偷修改 `isReady`。
- 不应该根据业务语义吞掉错误码。

一旦 adapter 开始做这些事，core 和 port 的职责就混了。

## 5. 默认配置与默认映射

当前默认配置放在 `gMpu6050PortDefCfg` 中，这说明下面这些值被视为工程级默认值：

- 默认 IIC 类型。
- 默认 bus。
- 默认地址高低位选择。
- 默认采样率、DLPF、量程。

这样做的好处：

- core 不需要硬编码当前板子的连接方式。
- 同一个 core 可在其他工程中只改 port 层就复用。
- 多个逻辑设备的默认映射集中在一个地方维护。

## 6. 初始化流程

当前 `mpu6050Init()` 的顺序可以直接视为该模块的标准 bring-up 契约：

1. 获取设备上下文。
2. 校验设备和配置是否合法。
3. 校验绑定和接口表是否合法。
4. 调用 `iicIf->init()` 初始化底层总线。
5. 清 `isReady`，清缓存。
6. 读取 `WHO_AM_I`。
7. 判断设备 ID 是否兼容。
8. 写复位寄存器。
9. 调 `mpu6050PortDelayMs()` 等待复位完成。
10. 下发时钟源、唤醒、采样率、滤波、量程配置。
11. 全部成功后置 `isReady = true`。

这个流程说明：

- port 只提供原子底层动作。
- 初始化顺序完全由 core 决定。
- delay 钩子虽然在 port 层，但何时等待由 core 决定。

## 7. 公共 API 使用顺序

推荐按下面顺序使用：

1. 调用 `mpu6050GetDefCfg(device)` 装载默认配置。
2. 如果要改绑总线，先修改 `stMpu6050Cfg` 里的 `iicBind`。
3. 调用 `mpu6050Init(device)`。
4. 用 `mpu6050IsReady(device)` 判断模块是否可用。
5. 再调用 `mpu6050ReadRaw()`、`mpu6050ReadTempCdC()` 等业务接口。

如果后续需要扩展 `GetCfg()` / `SetCfg()`，也应保持这个顺序不变。

## 8. 错误处理与 ready 约束

当前模块的状态约束非常明确：

- 配置或绑定非法，返回 `INVALID_PARAM`。
- 绑定合法但接口表不完整，返回 `NOT_READY`。
- 总线初始化失败，直接返回底层状态。
- 设备 ID 不匹配，返回 `DEVICE_ID_MISMATCH`。
- 只有初始化全部成功后，`isReady` 才能为 true。

这意味着后续扩展这个模块时，不能为了“方便调用”而绕过 `isReady` 检查。

## 9. 后续扩展点

如果后面还要继续扩展 MPU6050 模块，建议沿着当前契约扩展：

- 新增业务接口可以放在 core。
- 新增总线实现可以在 port 层增加新的 adapter 和接口表项。
- 新增默认映射可以扩展 `gMpu6050PortDefCfg`。
- 新增平台等待方式可以扩展 `mpu6050PortDelayMs()`。

但无论怎么扩展，都应保持一个原则：

- core 决定业务语义。
- port 只负责把项目里的 drv 能力链接给 core。
