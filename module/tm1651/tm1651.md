# TM1651 模块设计说明

## 1. 模块目标

`tm1651` 是一个数码管显示模块。它对上提供初始化、亮度控制、显示开关和常用数字显示接口；对下通过 `tm1651_port.c` 适配 `drvanlogiic`，复用项目里的软件两线总线绑定。

当前工程默认把 `TM1651_DEV0` 绑定到 `DRVANLOGIIC_TM`，对应板级引脚为：

- `PD2` -> `CLK`
- `PC12` -> `DIO`

## 2. 协议定位

TM1651 旧驱动虽然使用两线时序和 ACK，但它不是标准 I2C 寄存器读写模型：

- 没有标准 7bit 从地址阶段。
- 命令字节和地址字节由模块自己组织。
- 数据按 LSB first 发送。

因此本模块没有直接使用 `drvAnlogIicWriteRegister()` 之类的高层事务接口，而是通过 `drvanlogiic` 新增的受锁保护 raw bus action 入口，在 port 层实现 TM1651 的单帧写入适配。

## 3. 文件分工

### 3.1 `tm1651.h`

负责定义：

- 逻辑设备编号 `eTm1651MapType`
- 命令宏、符号宏、状态码宏
- 公共稳定配置结构 `stTm1651Cfg`
- 仅供装配期使用的 assembly 契约由 `tm1651_assembly.h` / `tm1651_port.h` 提供
- 配置结构 `stTm1651Cfg`
- 运行时结构 `stTm1651Device`
- 对外公共 API

### 3.2 `tm1651.c`

负责实现：

- 默认配置懒加载
- 配置合法性检查
- 初始化流程
- 符号到段码的编码
- 显示刷新和显示控制
- 常用三位数字/错误码显示接口

core 只依赖 port 提供的两个原子动作：

- `init`
- `writeFrame`

### 3.3 `tm1651_port.h`

负责声明：

- 默认稳定配置接口
- assembly helper 和默认绑定接口
- assembly helper
- 绑定合法性检查接口
- 板级默认设备包装接口

### 3.4 `tm1651_port.c`

负责实现：

- 默认配置数组
- `DRVANLOGIIC_TM` 绑定
- TM1651 LSB-first 单帧发送 adapter
- 默认设备 `TM1651_DEV0` 的便捷调用包装

## 4. 初始化流程

`tm1651Init()` 按下面顺序执行：

1. 获取设备上下文并校验配置。
2. 初始化底层软件两线总线。
3. 清空显示缓存。
4. 发送数据模式命令、地址数据帧和显示控制命令。
5. 所有步骤成功后置 `isReady = true`。

## 5. 默认配置

当前默认值放在 `gTm1651PortDefCfg` 中：

- 默认总线：`DRVANLOGIIC_TM`
- 默认亮度：`7`
- 默认有效位数：`3`
- 默认上电显示使能：`true`

这里把 `digitCount` 默认成 `3`，是为了保持与旧工程 `0xC0`、`0xC1`、`0xC2` 三个位地址的实际用法一致；如果后续板级需要第四位，可以只在 port 默认配置里调整。

## 6. 公共 API 使用顺序

推荐按下面顺序使用：

1. 准备 `stTm1651Cfg cfg`，调用 `tm1651GetDefCfg(device, &cfg)` 取得稳定默认配置。
2. 如需覆写亮度、位数或显示开关，修改 `cfg` 后调用 `tm1651SetCfg(device, &cfg)`。
3. 如需调整底层总线绑定，准备 `stTm1651PortAssembleCfg assembleCfg`，调用 `tm1651PortGetDefAssembleCfg(device, &assembleCfg)` 后再用 `tm1651PortAssembleSoftIic()` 或 `tm1651PortSetAssembleCfg()` 完成装配。
4. 调用 `tm1651Init(device)`。
5. 用 `tm1651IsReady(device)` 判断模块是否可用。
6. 再调用亮度、显示开关和显示输出接口。

## 7. 生命周期契约

当前 `tm1651` 归类为 `passive module`，契约如下：

- cfg ownership：调用者持有临时 `stTm1651Cfg`；底层总线绑定由 `stTm1651PortAssembleCfg` 单独持有并通过 port API 管理。
- ready 条件：总线初始化、显示缓存刷新和显示控制命令全部成功后才允许 ready。
- repeat init：允许，等价于基于当前 cfg 重新完成一次显示器 bring-up。
- hot reconfig：允许，但必须先 `SetCfg()`，再重新 `Init()`。
- recover path：显示链路异常后，按 `GetDefCfg/GetCfg -> SetCfg -> Init` 恢复。
